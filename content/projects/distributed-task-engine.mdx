## Problem

For a systems course project, I wanted to understand distributed task processing from first principles — not just use Celery or Bull, but build one. The goal was a system that could accept HTTP-submitted jobs, distribute them across workers, handle failures gracefully, and provide real-time visibility into job state.

## Constraints

- Must support at-least-once delivery semantics (no silent job drops)
- Workers had to be horizontally scalable — spin up N workers and throughput should scale linearly
- The API server had to handle bursty traffic (1,000 jobs submitted in 1 second) without dropping requests
- Built as a solo project in 5 weeks alongside other coursework
- Had to be deployable with a single `docker compose up`

## Approach & Architecture

```
                    ┌─────────────────────┐
                    │   REST API Server   │
                    │   (Go + Gin)        │
                    │   POST /jobs        │
                    │   GET  /jobs/:id    │
                    └──────────┬──────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
        ┌─────▼─────┐  ┌──────▼──────┐  ┌──────▼──────┐
        │  Redis     │  │  PostgreSQL │  │  WebSocket  │
        │  (queue +  │  │  (job state │  │  (live      │
        │   pub/sub) │  │   + results)│  │   updates)  │
        └─────┬─────┘  └─────────────┘  └─────────────┘
              │
    ┌─────────┼─────────┐
    │         │         │
┌───▼───┐ ┌──▼────┐ ┌──▼────┐
│Worker │ │Worker │ │Worker │    (N instances)
│  #1   │ │  #2   │ │  #N   │
└───────┘ └───────┘ └───────┘
```

**Job lifecycle**: Client submits a job via POST → API server writes to PostgreSQL (PENDING) and pushes to Redis queue → Worker pops from queue, sets status to RUNNING, executes, writes result back → On failure, job is re-queued with exponential backoff up to 3 retries → Dashboard shows real-time state via WebSocket.

Key design decisions:

- **Redis for queuing, Postgres for state**: Redis gives fast pop semantics; Postgres gives durability and queryability. Jobs are only ACK'd from Redis after Postgres confirms the result write — this is what gives at-least-once guarantees.
- **Visibility timeout**: If a worker crashes mid-job, a background reaper process detects stale RUNNING jobs (no heartbeat for 30s) and re-queues them.
- **Idempotency keys**: Clients can attach an idempotency key to prevent duplicate job creation from retried HTTP requests.

## My Contributions

- Designed and built the entire system solo: API server, worker pool, reaper, and dashboard
- Wrote the Go API server (~1,800 lines) using Gin, with structured logging, graceful shutdown, and health checks
- Implemented the worker supervisor pattern: a parent process manages child worker goroutines, handles signals, and restarts crashed workers
- Built the React dashboard with real-time WebSocket updates showing job counts, worker status, and throughput graphs
- Wrote a load testing harness that can simulate bursty and sustained workloads

## Results

- **Throughput**: 10,400 jobs/hour sustained with 4 workers (CPU-bound test jobs averaging 1.2s each)
- **Scaling**: Near-linear throughput scaling up to 8 workers (measured 7.6x at 8 workers)
- **Reliability**: Zero job drops across 50,000 test jobs, including simulated worker crashes and network partitions
- **Latency**: p50 queue-to-start latency of 12 ms, p99 of 48 ms under sustained load
- **Cold start**: Full system up and serving in < 5 seconds with `docker compose up`
